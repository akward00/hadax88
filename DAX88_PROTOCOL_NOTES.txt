Dayton Audio DAX88 / Matrio TCP Protocol Notes
Working notes from the debug GUI and packet captures

Document status
===============

This is not an official protocol specification. It is a working description of
what we have learned while comparing:

- The Dayton Audio DAX88 user manual command model.
- The Matrio Control mobile app network traffic.
- The local DAX88 debug GUI implementation.
- Live behavior from the amplifier at TCP port 8899.

The command ids and value encodings line up with the DAX88-style matrix control
model: zones, sources, volume, bass, treble, balance, power, and mute. The TCP
framing and pushed status behavior are Matrio/app behavior that the manual does
not clearly document in the copy we have.

Important caveat
================

We do not yet fully understand how every status byte maps back to every logical
zone. Commands use the logical zone number directly and work. Some status blocks
appear to use a different slot order, especially for power. That is why the debug
GUI currently treats uncalibrated power status as unknown until a pushed event and
the following status block prove which raw status slot belongs to that logical
zone.

Transport
=========

Protocol transport:

- TCP
- Default port: 8899
- The app opens a persistent TCP connection.
- On connect, the app sends a startup query.
- After the query, the amplifier sends pushed frames on the same connection.

Observed startup behavior:

1. Client opens TCP connection to the DAX88.
2. Client sends the query payload.
3. DAX88 replies with config/name and status frames.
4. DAX88 continues to push command/event/status/config frames while the connection
   remains open.

This looks more like an implicit subscription than classic polling. The query is
the startup request; the open socket is the subscription.

Outer TCP frame
===============

Observed frames use this wrapper:

Offset  Size  Description
------  ----  -----------
0       4     Magic/header: 18 96 18 20
4       4     Payload length, little endian uint32
8       2     Checksum, little endian uint16
10      10    Reserved/zero bytes observed as 00 00 00 00 00 00 00 00 00 00
20      N     Payload

Checksum:

- Sum all payload bytes.
- Keep the low 16 bits.
- Store little endian at offset 8.

In Python terms:

    checksum = sum(payload) & 0xffff
    frame = magic + len(payload).to_bytes(4, 'little') + checksum.to_bytes(2, 'little') + ten_zero_bytes + payload

Common payload prefix
=====================

Most observed payloads start with ASCII:

    MCU+PAS+

Hex:

    4d 43 55 2b 50 41 53 2b

After the prefix, observed payloads use byte 82 followed by a command/type byte.

Startup query
=============

Payload:

    MCU+PAS+ 82 0a ff ff ff 89 26

Hex payload:

    4d 43 55 2b 50 41 53 2b 82 0a ff ff ff 89 26

Full observed/built frame:

    18 96 18 20 0f 00 00 00 57 06 00 00 00 00 00 00 00 00 00 00 4d 43 55 2b 50 41 53 2b 82 0a ff ff ff 89 26

This is the frame the Matrio app sends at startup. The DAX88 then returns config
and status information.

Command frames
==============

Command payload shape:

    MCU+PAS+ 82 <command_id> <value> <zone_mask_8_bytes> ff cc 26

The zone mask is eight bytes, one byte per zone:

- 01 means the target zone.
- 02 means not targeted.

Example target mask for zone 4:

    02 02 02 01 02 02 02 02

Observed command ids
====================

Command    ID hex  Notes
-------    ------  -----
volume     01      Display 0..38, raw value is display + 1
treble     02      Display -12..+12, raw value is display + 13
bass       03      Display -12..+12, raw value is display + 13
balance    05      Display 0..20, raw value is 1 + display * 3
power      08      off = 01, on = 02
source     0d      Source number 1..8
mute       0e      unmuted = 01, muted = 02

Examples
========

Zone 4 power on:

    18 96 18 20 16 00 00 00 ab 04 00 00 00 00 00 00 00 00 00 00 4d 43 55 2b 50 41 53 2b 82 08 02 02 02 02 01 02 02 02 02 ff cc 26

Zone 4 power off:

    18 96 18 20 16 00 00 00 aa 04 00 00 00 00 00 00 00 00 00 00 4d 43 55 2b 50 41 53 2b 82 08 01 02 02 02 01 02 02 02 02 ff cc 26

Zone 4 source 3:

    18 96 18 20 16 00 00 00 b1 04 00 00 00 00 00 00 00 00 00 00 4d 43 55 2b 50 41 53 2b 82 0d 03 02 02 02 01 02 02 02 02 ff cc 26

Config/name frames
==================

Payload type:

    MCU+PAS+ 82 15 ...

After 82 15, the payload contains length-prefixed strings:

    <len> <bytes> <len> <bytes> ... terminator

Observed order:

1. Device name
2. Zone 1 name
3. Zone 2 name
4. Zone 3 name
5. Zone 4 name
6. Zone 5 name
7. Zone 6 name
8. Zone 7 name
9. Zone 8 name
10. Source 1 name
11. Source 2 name
12. Source 3 name
13. Source 4 name
14. Source 5 name
15. Source 6 name
16. Source 7 name
17. Source 8 name, if present

Example observed names from this system:

- Device: 957 house audio
- Zones: Basement, Family Room, Terrace, Office, Pool, Hearth Room, ZONE7, ZONE8
- Sources: Songbird/4Stream, Sonos, Patio TV, ofc BT AD-BA010, Input5, Input6, Input7

Status frames
=============

Payload type:

    MCU+PAS+ 82 0c ...

After 82 0c, the first 56 bytes appear to be seven groups of eight bytes. The
current parser interprets them as:

Group  Index range  Current guess      Display conversion
-----  -----------  -------------      ------------------
0      0..7         source_raw          source number 1..8
1      8..15        volume_raw          display = raw - 1
2      16..23       treble_raw          display = raw - 13
3      24..31       bass_raw            display = raw - 13
4      32..39       balance_raw         display = (raw - 1) / 3 when valid
5      40..47       power_raw           guessed 01 = off, 02 = on
6      48..55       mute_raw            guessed 01 = unmuted, 02 = muted

Known problem with status frames
================================

The group interpretation is not proven. Earlier we thought the field groups were
broadly right and only the slot-to-zone mapping was wrong. A later startup test
showed all zones physically off while the parsed startup status still produced
power_raw = 02 for raw slot 1. That means at least one of these assumptions is
wrong: the field offset, the power value meaning, the status-frame type, or the
slot-to-zone mapping.

For example, the raw power group can show slot 1 as on even when Matrio and the
real system say all logical zones are off. We also observed Office power events
where the immediate pushed event correctly identified Office, then a later status
block appeared to update a different raw slot.

What we know:

- Command zone masks are logical and work as expected.
- Event frames identify the logical zone correctly.
- Status frames contain repeatable structured data, but our field interpretation
  is not yet authoritative.
- The raw slot inside a status group may not equal the logical zone number.
- Power is specifically not solved: a startup all-off state has been observed with
  bytes that our current parser would interpret as a powered-on raw slot.

Current debug GUI workaround:

- Do not trust uncalibrated power status at startup.
- When a pushed event says logical zone N changed field F to value V, remember the
  last confirmed status block as a baseline.
- When the next status block arrives, compare baseline vs new status.
- If exactly one raw slot changed to value V for field F, learn:

      logical zone N / field F -> raw status slot S

- Persist that learned map in browser localStorage.

Pushed event frames
===================

Event frames use the same command payload shape as command frames:

    MCU+PAS+ 82 <command_id> <value> <zone_mask_8_bytes> ...

The Matrio app appears to receive these on the persistent TCP connection. These
frames are useful because they include the logical zone mask. In practice, they
are often more reliable for immediate UI updates than the following status block,
because the status block may use a raw slot order we have not mapped yet.

Observed Office example:

- Office source changed to Patio TV: event mask pointed to zone 4 and source value
  was applied correctly.
- Office power on/off: event mask pointed to zone 4, but the later status block
  caused confusion until we stopped assuming status slot 4 always means logical
  zone 4.

Unknown frames
==============

The parser classifies a frame as unknown when:

- The outer frame checksum is invalid or no valid payload can be extracted.
- The payload does not match known config, status, or event shapes.
- The payload starts like a command/event but the command id is not one we know.

The debug GUI keeps the last unknown frame and shows it in the status area/debug
data so we can capture it for later protocol work.

RS232 manual command mapping
============================

The DAX88 manual documents an RS232 command form:

    <xxPPuu\r

Where:

- xx is the zone number, 01..08.
- PP is the action code.
- uu is the command value.
- CR is carriage return, 0d.

Manual RS232 actions compared to observed TCP commands:

Manual  Meaning   Manual value            TCP command id  TCP value
------  -------   ------------            --------------  ---------
PR      Power     00 off, 01 on           08              01 off, 02 on
MU      Mute      00 off, 01 on           0e              01 off, 02 on
VO      Volume    00..38                  01              manual value + 1
TR      Treble    00..24 => -12..+12      02              manual value + 1
BS      Bass      00..24 => -12..+12      03              manual value + 1
BL      Balance   00..20                  05              appears to be 1 + value * 3
CH      Source    01..08                  0d              same value, 01..08

So the action set maps well. TCP appears to carry the same logical controls inside
a binary Matrio wrapper, with command ids instead of ASCII action codes and mostly
one-based raw values.

Examples:

- Manual <01PR00 means zone 1 power off.
- TCP equivalent is command id 08, value 01, zone mask slot 1 = 01.
- Manual <01PR01 means zone 1 power on.
- TCP equivalent is command id 08, value 02, zone mask slot 1 = 01.
- Manual <06CH04 means zone 6 source 4.
- TCP equivalent is command id 0d, value 04, zone mask slot 6 = 01.

RS232 manual status mapping
===========================

The manual documents status query:

    ?xx\r

Where xx is one zone, 01..08.

Manual reply shape:

    >xxaabbccddeeffgghhiijj\r

Manual fields:

Field  Meaning
-----  -------
aa     PA control status
bb     Power control status
cc     Mute control status
dd     DT control status
ee     Volume control status
ff     Treble control status
gg     Bass control status
hh     Balance control status
ii     Source control status
jj     Keypad connection status, 00 not connected, 01 connected

This is important because it does not match our first TCP status parser cleanly.
Our parser assumed the 82 0c TCP status bytes were grouped as source, volume,
treble, bass, balance, power, mute. The manual status order suggests a different
possibility: TCP status may be a compact all-zone version of the RS232 per-zone
status reply, or it may include only selected RS232 fields but in manual order.

This could explain the startup bug: all physical zones were off, but our parser
looked at one byte as power and interpreted it as on. That byte may not be power
at all, or TCP status may encode power differently from command values.

Current interpretation after comparing the manual
=================================================

The command path maps cleanly enough to continue using it:

- Zone targeting is logical and matches manual zones 01..08.
- Power, mute, volume, treble, bass, balance, and source all have obvious manual
  equivalents.
- TCP command values are mostly the manual values shifted by +1, except source
  stays 1..8 and balance appears scaled.

The status path does not yet map cleanly:

- The manual exposes a per-zone ASCII status query.
- The Matrio TCP stream exposes binary pushed status blocks.
- We have not proven how the binary status block maps to the manual aa..jj fields.
- We should treat TCP command/event frames as better understood than TCP status
  frames until we capture enough known ground-truth transitions.

Current conclusions
===================

Reasonable confidence:

- TCP port 8899 is the Matrio/DAX88 control path.
- The outer frame wrapper, payload length, and checksum are understood.
- The startup query is understood.
- Config/name frames are understood.
- Command frame construction is understood for source, power, mute, volume, bass,
  treble, and balance.
- Event frames identify logical command changes.
- Status block framing is detected, but field meanings and slot mapping are not
  yet fully understood.

Not yet fully understood:

- The authoritative meaning of each status group and byte offset.
- The authoritative mapping from status slots to logical zones for each field.
- Whether status slot order can vary by field, by firmware, by grouping, or by
  amplifier/expansion topology.
- Whether there is another official query that returns the slot map.
- Whether the manual's RS-232/control section describes this status slot mapping
  in a way we have not extracted yet from the PDF.

Recommended next debugging steps
================================

1. Start with all zones physically off and record a baseline status block. Mark
   this as ground truth: every logical zone is off, regardless of parsed bytes.
2. Turn on exactly one logical zone from Matrio.
3. Save the event frame and the next status frame.
4. Repeat for every zone.
5. Build a table:

      logical zone -> raw power status slot

6. Repeat for source, mute, volume, bass, treble, and balance if needed.
7. Once the slot map is known, move this logic from the debug GUI into the Home
   Assistant integration.

Home Assistant implication
==========================

The current Home Assistant integration should not be considered final until the
status-slot mapping problem is solved. The command path can work while the status
path is still wrong, which leads to dangerous UI behavior: Home Assistant may show
one zone on/off based on another zone's raw status slot.

The integration should eventually use the same persistent subscription model as
Matrio, with event frames as primary updates and status blocks as confirmation.
Until then, power state should be treated carefully.
